# Recommended Sections

Also known as the [Diátaxis](https://diataxis.fr/) approach

Documentation should be made up of four sections
1. Tutorials (Learning-Oriented): Lessons that take the reader by the hand through a series of steps to complete a project
2. How-to Guides (Problem-Oriented): Guides that take the reader through the steps required to solve a common problem
3. Discussions (Understanding-Oriented): Explanations that clarify or illuminate a particular topic or design decision
4. Reference (Information-Oriented): Technical descriptions of the machinery and how to operate it
![[Media/diagram_documentation.png]]

## Tutorials

These are focused on new learners, and should be able to follow the tutorial from beginning to end without any problems.
- Getting started
- Immediate sense of achievement
- Concreteness, not abstraction
- Minimum necessary explanation and minimal working examples
- Inspire confidence in the software and their own abilities
- Must work and be repeatable

## How-to Guides

These are problem oriented guides. The user may have these problems, whereby you provide the solution.
- A series of steps
- Focus on the goal
- Addressing a particular question
- Some flexibility/abstraction to solve a general problem
- Good naming

## References

These have one job to describe, things like key classes, functions and APIs. This is information oriented and not necessarily goal oriented.
- Structure - Should match the code base
- Consistency - Should be consistent across all parts of the documentation
- Description
- Accuracy - Could be tightly coupled to source to ensure it's up to date

## Discussions

- Giving context
- Explaining why
- Multiple examples, alternative approaches (perhaps ones that the user may already use)
- Making connections
- No instructions or technical description
