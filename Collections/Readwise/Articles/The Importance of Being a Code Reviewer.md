## Highlights

- **Keep personal preferences optional
  **When you suggest changes in code reviews, make sure that these changes are real improvements to the code and not the reviewer’s personal preferences, which are not necessarily the same as the company or industry best practices. I use this formula when I propose such a change: *“This is a matter of personal preference, but if you like it <code change>”* ([View Highlight](https://read.readwise.io/read/01h7k6j42te1dcme3rh0bk7vy8))
- **Compliment the author
  *“****LGTM”* (looks good to me) *might be interpreted as “I reviewed your PR quickly”.* Personally, I’m ok when I read that, however, don’t hesitate to write down in the PR how well the code has been written (If you genuinely think it is true!). Here are a couple of reasons why a praise is well deserved: a new fancy functionality has been introduced, or a complex yet well-architected refactoring occurred.
